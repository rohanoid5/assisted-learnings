# Upgrade Procedures

## Concept

PostgreSQL major versions (e.g., 15 → 16) change the internal data format, so you cannot simply replace binaries. You must use a proper upgrade approach. Three main strategies exist: **pg_upgrade** (fastest, zero downtime is not built-in), **logical replication** (near-zero downtime, covered in Module 09-03), and **dump/restore** (simplest but slow for large databases). Choosing the right approach depends on your database size, acceptable downtime window, and whether you can run two clusters in parallel.

---

## Upgrade Strategy Comparison

| Strategy | Downtime | Database Size | Complexity |
|----------|----------|---------------|------------|
| `pg_dump` / `pg_restore` | Hours (data-dependent) | Small (< 50GB) | Low |
| `pg_upgrade` | Minutes | Any size | Medium |
| Logical replication | Seconds | Any size | High |
| pg_upgrade + streaming | Minutes + failover | Any size | Medium |

## Pre-Upgrade Checklist

```sql
-- 1. Check current version:
SELECT version();
SHOW server_version;

-- 2. Verify your extensions are compatible with the target version:
SELECT name, default_version, installed_version
FROM pg_available_extensions
WHERE installed_version IS NOT NULL
ORDER BY name;
-- Check extension release notes for each: pg_trgm, pgcrypto, uuid-ossp are bundled
-- Third-party: pg_cron, pg_stat_statements may need binary updates.

-- 3. Check for deprecated features (review PostgreSQL release notes):
-- PG 15: removed query_id, changed COPY behavior
-- PG 16: removed --sync-standby-names, added pg_stat_io

-- 4. Document all custom postgresql.conf settings:
SELECT name, setting, unit, source
FROM pg_settings
WHERE source != 'default'
ORDER BY name;

-- 5. Check disk space — pg_upgrade needs space for the new cluster alongside the old:
-- df -h /var/lib/postgresql/

-- 6. Note all cron jobs, connection strings, pg_hba.conf entries to migrate.
```

## Method 1: pg_dump / pg_restore

Best for smaller databases (< 50GB) or when you want the simplest approach:

```bash
# Step 1: Stop write traffic to the old database (put app in maintenance mode).

# Step 2: Take a final fresh backup:
pg_dump -h localhost -U storeforge -d storeforge_dev \
    --format=custom --jobs=4 --file=pre_upgrade.dump

# Step 3: Install PG 16 binaries.

# Step 4: Init a new PG 16 cluster:
/usr/lib/postgresql/16/bin/initdb -D /var/lib/postgresql/16/main

# Step 5: Create the database:
/usr/lib/postgresql/16/bin/pg_ctl start -D /var/lib/postgresql/16/main
createdb -h localhost -p 5433 -U postgres storeforge_dev

# Step 6: Restore:
pg_restore -h localhost -p 5433 -U postgres \
    --dbname=storeforge_dev \
    --jobs=4 \
    pre_upgrade.dump

# Step 7: Update connection strings and start application.
```

## Method 2: pg_upgrade (In-Place Upgrade)

`pg_upgrade` upgrades data files in-place without re-importing data — typically 10× faster than dump/restore for large databases:

```bash
# Step 1: Install PG 16 binaries alongside PG 15.
# Step 2: Stop PG 15:
pg_ctlcluster 15 main stop
# or: pg_ctl stop -D /var/lib/postgresql/15/main

# Step 3: Initialize a new PG 16 data directory:
/usr/lib/postgresql/16/bin/initdb \
    --pgdata=/var/lib/postgresql/16/main \
    --encoding=UTF8 \
    --data-checksums

# Step 4: Run pg_upgrade in CHECK mode first (dry run — no changes):
/usr/lib/postgresql/16/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/15/main \
    --new-datadir=/var/lib/postgresql/16/main \
    --old-bindir=/usr/lib/postgresql/15/bin \
    --new-bindir=/usr/lib/postgresql/16/bin \
    --check \
    --verbose
# Verify: "Clusters are compatible" must appear before proceeding.

# Step 5: Run the actual upgrade:
/usr/lib/postgresql/16/bin/pg_upgrade \
    --old-datadir=/var/lib/postgresql/15/main \
    --new-datadir=/var/lib/postgresql/16/main \
    --old-bindir=/usr/lib/postgresql/15/bin \
    --new-bindir=/usr/lib/postgresql/16/bin \
    --link \        # hardlinks instead of copying → much faster (in-place)
    --verbose
# --link mode: upgrade completes in seconds; old cluster data files are shared.
# ⚠️  With --link, starting the old cluster again will corrupt both clusters.
# Without --link: files are copied — slower but old cluster remains intact.

# Step 6: Copy pg_hba.conf and postgresql.conf from old to new:
cp /var/lib/postgresql/15/main/pg_hba.conf /var/lib/postgresql/16/main/
cp /var/lib/postgresql/15/main/postgresql.conf /var/lib/postgresql/16/main/
# Update postgresql.conf to remove settings that changed in PG 16.

# Step 7: Start PG 16:
pg_ctl start -D /var/lib/postgresql/16/main

# Step 8: Update statistics (required after pg_upgrade — stats are not copied):
/usr/lib/postgresql/16/bin/vacuumdb --all --analyze-in-stages \
    -h localhost -U storeforge

# Step 9: Run the cleanup script generated by pg_upgrade:
./delete_old_cluster.sh   # generated by pg_upgrade, removes PG 15 data
```

## Method 3: Logical Replication (Near-Zero Downtime)

See Module 09-03 for the full walkthrough. Summary:

```bash
# 1. Install PG 16 on a new server.
# 2. Restore schema-only dump to PG 16.
# 3. Create a publication on PG 15: CREATE PUBLICATION upgrade_pub FOR ALL TABLES;
# 4. Create a subscription on PG 16 pointing at PG 15.
# 5. Wait for initial sync + streaming to catch up (lag_bytes ≈ 0).
# 6. Redirect app traffic to PG 16 (update PgBouncer target).
# 7. Drop subscription on PG 16. Done.
# Downtime: only the seconds it takes to redirect PgBouncer.
```

## Post-Upgrade Validation

```sql
-- 1. Verify all tables are accessible:
SELECT count(*) FROM product;
SELECT count(*) FROM "order";
SELECT count(*) FROM order_item;

-- 2. Check extensions re-installed correctly:
SELECT name, installed_version FROM pg_available_extensions
WHERE installed_version IS NOT NULL;

-- 3. Verify sequences are correct:
SELECT sequence_name FROM information_schema.sequences WHERE sequence_schema = 'public';
SELECT setval('order_id_seq', (SELECT MAX(id) FROM "order"));  -- if sequences reset

-- 4. Check pg_upgrade's log for warnings:
-- cat /tmp/pg_upgrade_output.d/pg_upgrade_server.log

-- 5. Run VACUUM ANALYZE (pg_upgrade does not carry over statistics):
VACUUM ANALYZE VERBOSE;

-- 6. Check for incompatible query patterns by reviewing pg_stat_statements.
```

```sql
-- 1. Version check:
SELECT version();
-- Returns full version string: "PostgreSQL 16.3 on x86_64-pc-linux-gnu..."

SHOW server_version;
-- Returns just the version number: "16.3"

SHOW server_version_num;
-- Returns integer: 160003 (useful for version comparisons in scripts)

-- 2. Hardlinks flag:
-- --link flag creates hardlinks (not copies) between old and new data directories.
-- Tradeoff:
-- PRO:  Upgrade completes in seconds (no data copying for large databases)
-- CON:  If you start the OLD cluster after upgrade, files are corrupted (shared inodes)
-- Use --link only when you are confident in the upgrade and have a backup.

-- 3. ANALYZE after pg_upgrade:
-- pg_upgrade does NOT migrate planner statistics (pg_statistic table is cleared).
-- Without statistics, the planner has no column distribution data → makes very
-- poor execution plan choices (treats every table as empty).
-- --analyze-in-stages runs ANALYZE in three passes at increasing fidelity,
-- so the planner has usable statistics as fast as possible.

-- 4. Strategy ranking for 200GB database:
-- Downtime (shortest first):
-- 1. Logical replication: seconds (only PgBouncer redirect)
-- 2. pg_upgrade with --link: minutes (write stop + pg_upgrade + vacuumdb --analyze)
-- 3. pg_restore --jobs=4: hours (4 parallel workers * 200GB restore)

-- Complexity (simplest first):
-- 1. pg_restore: lowest — well-known tools, straightforward
-- 2. pg_upgrade: medium — binary version management, file paths
-- 3. Logical replication: highest — schema sync, version compatibility, monitoring replication

-- Rollback ease (easiest first):
-- 1. pg_restore: trivial — old cluster untouched, just delete the new one
-- 2. pg_upgrade (without --link): old cluster intact, start it back up
-- 3. pg_upgrade (with --link): very hard — hardlinks shared, old cluster corrupted
-- 3. Logical replication: easy during sync; harder after cutover (re-add subscription)
```

## Capstone Connection

StoreForge upgrade procedure:
- **Quarterly minor updates** (e.g., 16.1 → 16.3): done with `apt upgrade postgresql-16` + `pg_ctlcluster restart` — no data migration needed
- **Major version upgrade** (15 → 16): used logical replication approach — 72 hours of parallel operation, 18 seconds of actual application downtime during PgBouncer reconfiguration
- **Pre-upgrade checkout**: always run `pg_upgrade --check` first; address all warnings before the production window
- **Rollback plan**: during the logical replication upgrade, the old PG 15 cluster was left running for 48 hours post-cutover as a fallback — then decommissioned after confirming zero regressions
